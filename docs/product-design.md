# 项目设计文档

## 1. 项目概述

### 1.1 项目名称

**集盒FolioBox**
**把零散项目整理成更能投的作品集**

### 1.2 一句话定义

一款面向国内设计师的 **设计稿读取 + 简历信息导入 + 项目事实补充 + AI 叙事生成 + 作品集评分诊断** 工具，帮助用户把分散在设计平台中的真实项目，快速整理成 **可投递、可分享、可继续编辑** 的作品集页面与 PDF，并逐步延伸到岗位匹配与面试讲述准备。

### 1.3 项目要解决的问题

大量设计师并不是没有项目，而是缺少把项目整理成“可通过初筛”的作品集能力，具体表现为：

- 项目分散在设计工具中，素材整理成本高。
- B 端 / G 端 / 企业系统项目视觉不够“炸”，难包装。
- 不知道如何把项目背景、设计决策、个人价值讲清楚。
- 临近投递时，没时间从零排版、写文案、做导出。
- 面试时不会围绕作品集自然讲解，容易陷入“照稿念”。
- 不知道不同岗位、不同级别、不同方向的真实招聘要求是什么。

与此同时，面向国内大厂和成熟团队的求职场景里，作品集依然是设计师投递过程中的关键门槛。对于产品设计、UI 设计、UX 设计等岗位而言，作品集往往不只是“附加材料”，而是决定简历是否能进入下一轮沟通的重要依据。尤其在竞争激烈的招聘环境中，招聘方更关注的不是项目数量本身，而是候选人能否通过 3–5 个高质量案例，清楚呈现项目背景、个人角色、设计判断与实际价值。

### 1.4 核心定位

本产品不是“全自动作品集网站生成器”，而是：

> **一个帮助设计师基于真实项目事实，快速生成可投递 case study 初稿，并持续优化求职表达质量的工具。**

### 1.5 产品目标

帮助用户在 **10–20 分钟内** 生成一份 **80 分以上、可继续编辑、具备投递价值** 的作品集初稿，用于：

- 求职投递
- 客户提案
- 社交平台展示
- 后续个人品牌沉淀

### 1.6 长期愿景

长期来看，产品不应停留在“作品集工作台”，而应逐步成长为：

> **面向国内设计师的就业指导与求职表达平台。**

这意味着产品服务的终点，不只是“生成一份过关的作品集”，而是围绕设计师求职链路，持续覆盖以下关键环节：

1. **作品集生成与诊断**：帮助用户产出可投递、可过初筛的作品集
2. **简历与作品集对齐**：保证投递材料口径统一
3. **岗位要求映射**：让作品集表达与真实岗位需求更对齐
4. **面试讲述准备**：帮助用户围绕作品集进行更专业的项目讲解
5. **持续职业成长**：沉淀个人项目资产、表达策略与求职反馈

因此，当前的“作品集工作台”可以被视为第一阶段产品形态；中长期更合适的定位是：

> **设计师求职表达 Copilot / 设计师就业指导工作台**

## 2. 用户与场景定义

## 2.1 核心目标用户

### 第一优先级

**正在求职的产品 / UI / UX 设计师**。特征包括：

- 2–8 年经验
- 手里已有 2–5 个项目
- 主要项目存放在 Figma / 即时设计 / Pixso / MasterGo 等平台
- 缺少系统整理作品集的时间
- 希望提升初筛通过率，而不是单纯做一个“好看网站”

### 第二优先级

**自由设计师 / 独立接单设计师**。需求是快速包装项目，用于客户展示和报价沟通。

### 第三优先级

**企业内部设计师**。尤其是做 B 端 / G 端 / 复杂业务系统的设计师，希望把“不够亮眼但有价值”的项目包装成可讲清思路的案例。

### 第四优先级

**设计学生 / 转岗设计师**。需要快速产出结构清晰的作品集雏形。

---

## 2.2 用户痛点

### 痛点 1：整理成本高

- 页面很多，探索稿、废稿、方案稿混在一起。
- 不知道该选哪些画面展示。
- 导出、排版、长图、PDF 很花时间。

### 痛点 2：不会讲故事

- 只会展示结果图，不会写项目背景和设计价值。
- 不会提炼“为什么这样做”“我具体做了什么”。

### 痛点 3：项目不够“好看”

- B 端 / G 端 / 数据系统项目视觉克制，不适合直接裸放。
- 面试官未必能一眼看懂其复杂度与价值。

### 痛点 4：投递导向不明确

- 同一个项目投不同公司、不同岗位，重点应不同。
- 用户不知道应该突出视觉、逻辑、协作还是业务理解。

### 痛点 5：投后链路断裂

- 简历、作品集、面试讲述三者常常口径不一致。
- 用户即使做出作品集，也不知道如何围绕它进行专业表达。

## 2.3 轻量用户画像（MVP 版本）

### 画像 A：求职中的中级设计师（核心主画像）

#### 基本背景
- 2–5 年工作经验
- 做过 2–4 个完整项目
- 当前正在准备跳槽或提升岗位层级
- 平时项目分散在 Figma / 即时设计 / Pixso / MasterGo 中

#### 当前目标
- 尽快整理出一份能投递的作品集
- 让项目表达更接近国内大厂或成熟团队的期待
- 不希望从零排版和重写整套 case study

#### 核心痛点
- 有项目，但不会挑哪些页面展示
- 知道自己的项目有价值，但不会讲清楚
- 时间紧，不想在排版和导出上浪费太多精力

#### 最在意的结果
- 作品集“看起来够专业”
- 可以清楚表达自己的角色、判断和价值
- 真敢投，不会觉得拿不出手

#### 最容易卡住的地方
- 第一版生成是否足够过线
- 项目事实填写是否太重
- 作品集和简历是否前后不一致

#### 对应产品机会
- 免费评分入口
- 快速重制链路
- 多版本候选生成
- 简历与作品集对齐

### 画像 B：B 端 / G 端 / 企业系统设计师（差异化核心画像）

#### 基本背景
- 2–8 年经验
- 项目复杂、流程长、业务链条重
- 页面不一定“好看”，但真实价值很强
- 常担心作品集不够抓眼

#### 当前目标
- 把不够亮眼的项目讲成有说服力的 case study
- 让面试官快速看懂复杂度和价值

#### 核心痛点
- 视觉不够炸，直接展示容易显得平
- 很难解释项目复杂度、协作关系和业务影响
- 不知道怎么把系统型项目讲得既专业又不枯燥

#### 最在意的结果
- 项目被“看懂”而不是被忽略
- 复杂系统也能包装成可投递案例
- 不靠夸张视觉也能建立信任

#### 最容易卡住的地方
- 项目背景太复杂，不知道怎么压缩
- 结果数据不完整，不知道如何证明价值
- 保密信息不方便直接公开

#### 对应产品机会
- 评分诊断里的“角色 / 结果 / 价值”问题提示
- 更偏专业克制的风格轨道
- 保密与打码能力
- 面向 B 端岗位的表达优化

### 画像 C：求职窗口期高压用户（高支付意愿画像）

#### 基本背景
- 正在密集投递或已经拿到面试
- 时间非常紧
- 对结果和效率比对“慢慢打磨”更敏感

#### 当前目标
- 在最短时间内把最影响结果的问题修掉
- 尽快准备出一版能投、能讲的作品集

#### 核心痛点
- 不知道当前作品集到底差在哪
- 来不及从头做完整优化
- 面试快到了，讲述也没准备好

#### 最在意的结果
- 快速知道问题优先级
- 快速拿到更适合目标岗位的版本
- 在面试前把项目讲清楚

#### 最容易卡住的地方
- 评分之后不知道下一步干嘛
- 工具太复杂，学习成本太高
- 支付前不确定是否真的值得

#### 对应产品机会
- 免费评分 → 重制链路
- 求职冲刺版
- 面试讲述提纲
- 定向岗位优化

## 2.4 核心用户旅程地图（轻量版）

### 阶段 1：意识到作品集可能不过线

#### 用户目标
- 先判断自己当前作品集到底能不能投

#### 用户情绪
- 焦虑
- 不确定
- 有点抗拒重新做一遍

#### 核心阻碍
- 不知道问题在哪里
- 不愿意一开始就投入太多时间

#### 产品机会点
- 免费评分入口
- 首页案例展示与前后对比
- “更能投”而不是“更好看”的价值表达

### 阶段 2：进入产品并决定是否继续

#### 用户目标
- 尽量低成本地试一下
- 先确认这个产品值不值得继续用

#### 用户情绪
- 谨慎
- 试探
- 对 AI 结果存疑

#### 核心阻碍
- 担心流程太复杂
- 担心要填太多信息
- 担心项目保密和上传风险

#### 产品机会点
- 先评分，后注册或低门槛注册
- 最小资料收集
- 隐私与保密说明
- 案例页建立信任

### 阶段 3：进入重制链路

#### 用户目标
- 把真实项目更快整理成可投递版本

#### 用户情绪
- 期待
- 怕失控
- 希望系统帮忙，但又想保持可控

#### 核心阻碍
- 不知道哪些页面该选
- 不知道项目事实该写多细
- 怕 AI 一次生成一堆自己不认可的内容

#### 产品机会点
- 展示页确认
- 项目事实表单分步填写
- 先大纲确认，再成品渲染
- 2–3 个候选版本给用户选择

### 阶段 4：编辑与导出

#### 用户目标
- 把结果修到自己愿意投出去

#### 用户情绪
- 开始建立信心
- 但对细节仍然敏感

#### 核心阻碍
- 改动是否足够顺手
- 风格是否够专业
- PDF / 链接输出是否方便

#### 产品机会点
- 手动可视化编辑为主，AI 局部改写为辅
- 工作台式编辑器
- PDF / 链接导出
- 前后版本切换

### 阶段 5：投递前与面试前

#### 用户目标
- 确认这份作品集真的能用
- 如果进入面试，希望能讲清楚

#### 用户情绪
- 紧张
- 高度结果导向

#### 核心阻碍
- 不确定是否已经过线
- 不确定是否匹配目标岗位
- 不会围绕作品集自然表达

#### 产品机会点
- 详细评分与建议
- 岗位定向版本
- 求职冲刺版
- 面试讲述提纲（后续）

### 2.5 这套画像与旅程的使用方式

本项目中的用户画像和旅程地图，不作为厚重研究材料，而主要用于指导以下决策：

- 首页主叙事与案例展示优先服务谁
- 免费评分入口如何设计
- 项目事实表单应该收多重
- 哪一步必须手动确认，哪一步可以 AI 辅助
- 哪些能力适合放在高客单求职冲刺版中

## 3. 产品价值主张

### 3.1 对用户的核心价值

- **节省整理时间**：从简历与设计稿到作品集初稿，不再从零开始。
- **减少重复填写**：简历可作为基础资料入口，避免用户二次录入个人背景与经历。
- **提升表达质量**：AI 帮助把项目写成更适合招聘语境的 case study。
- **降低包装门槛**：尤其帮助 B 端 / G 端项目“被看懂”。
- **提高投递效率**：同一项目可生成不同岗位方向的版本。
- **提高通过初筛的概率**：不是保证拿面试，而是让作品集更接近招聘方的阅读期待。
- **延伸到面试准备**：围绕已生成的作品集，继续帮助用户建立更自然、更专业的项目讲述能力。
- **让作品集更贴近真实岗位需求**：结合国内大厂与成熟公司的真实招聘要求，帮助用户把表达重点放在真正被筛选的能力上。

### 3.2 产品原则

- **事实优先**：先收集真实背景与项目事实，再生成表达。
- **生成可编辑**：AI 生成的是初稿，不是不可修改成品。
- **投递导向**：围绕“能否被继续看下去”设计。
- **多输出形态**：链接、PDF、分享图都重要。
- **国内工作流优先**：不只服务“独立作品集网站”这一种习惯。
- **先做结果，再做功能堆叠**：优先解决“能不能投、能不能过筛、能不能讲清楚”。

### 3.3 优秀作品集的质量标准定义

“优秀作品集”在本项目中，不定义为单纯“页面好看”，而定义为：

> **能在有限浏览时间内，让招聘方快速建立专业信任，并清楚理解候选人的角色、判断与价值的作品集。**

本项目将“优秀作品集”的质量拆分为以下 8 个维度：

1. **首屏专业感**：第一眼像正式投递材料，而不是练习页或模板页。
2. **可扫描性**：招聘方可以在短时间内抓住项目背景、角色、亮点与结果。
3. **项目选择质量**：项目数量克制，优先展示 3–5 个高质量案例。
4. **角色清晰度**：清楚说明个人角色、参与深度、负责范围。
5. **问题定义与设计判断**：能说明问题是什么、为什么重要、为什么这样设计。
6. **结果与价值证明**：尽量提供结果、影响、反馈或替代证据。
7. **真实性与可信度**：不夸大角色，不编造结果，不使用明显套话。
8. **投递适配度**：内容与目标岗位相关，强调点与岗位要求一致。

### 3.4 作品集质量评分框架（MVP 内部标准）

建议按 100 分制拆分：

- 首屏专业感：15 分
- 可扫描性：15 分
- 项目选择质量：10 分
- 角色清晰度：15 分
- 问题定义与设计判断：20 分
- 结果与价值证明：15 分
- 真实性与可信度：5 分
- 投递适配度：5 分

建议定义质量等级：

- **85–100 分**：可直接用于正式投递
- **70–84 分**：已具备投递价值，但建议局部修改
- **50–69 分**：更适合作为草稿，需补信息或优化表达
- **50 分以下**：暂不建议投递

### 3.5 第一版生成质量的保障策略

考虑到设计师是对视觉和细节敏感度极高的群体，产品首版不能把“生成能力”建立在大量模板与风格自由度上，而要先保证“少而稳”，再给用户足够的选择空间。

#### 1）不是只做 1 套结果，而是一次生成 2–3 个候选版本

建议首发采用三条风格轨道：

- **专业克制版**：适合国内大厂投递、B 端 / G 端 / 企业平台项目
- **平衡展示版**：兼顾专业感与视觉表现，适合大多数通用投递场景
- **视觉表达版**：适合 C 端、品牌感更强、年轻设计师偏好的展示方式

#### 2）先保证首屏与阅读节奏

- 先把封面、项目概述、核心页面展示、亮点解析几部分打磨到位
- 不优先追求复杂交互、炫动效和高度个性化

#### 3）采用强约束生成，而不是自由生成

- 固定 case study 结构
- 固定模块顺序规则
- 固定标题层级与图文版式逻辑
- 降低“每次生成都长得不一样”的不稳定性

#### 4）视觉系统要分轨道，但都要过“专业感底线”

- 不同用户群可以接受不同程度的视觉表达强度
- 但所有版本都必须满足：层级清楚、阅读稳定、不过度模板化、不过度花哨
- 即使是“视觉表达版”，也不能牺牲可扫描性和招聘友好度

#### 5）把“质量诊断”做成产品能力

- 不是只生成结果
- 还要告诉用户：哪里不过关、哪里还空、哪里太像套话、哪里缺少结果证明
- 这样即使首版不能 100% 自动过线，也能通过诊断提高最终交付质量

### 3.6 作品集评分与诊断能力定义

该能力的核心定义是：

> **让用户先上传自己现有的作品集，快速获得一份可信的质量评分与诊断摘要，再引导其在平台内完成优化或重制。**

这项能力有三个价值：

- **降低首次使用门槛**：用户不必先完整导入项目，也能立即获得反馈。
- **增强价值感知**：用户先看见问题，再更愿意为“优化”和“重制”付费。
- **形成获客漏斗**：免费评分吸引流量，付费解锁深度建议与重制能力。

## 4. 竞争环境与市场机会

### 4.1 海外强邻居

当前海外已有几类成熟工具：

- **Figma** 已官方推出 AI portfolio website builder，并支持从项目展示到发布分享的流程。
- **Figma 插件生态** 已非常成熟，插件被官方定义为扩展编辑器工作流的重要机制。

这说明：

1. “作品集生成”不是伪需求。
2. 平台级玩家已经入场。
3. 单纯“建站”不是最好切口。

### 4.2 国内差异化机会

国内设计工具生态并不是 Figma 一统天下，而是存在明显的本土平台并存格局。

因此，本项目的机会不在于做“中国版 Framer”，而在于做：

> **更懂国内设计师投递环境、更懂复杂项目包装、更适合国内设计工具工作流的作品集生成产品。**

### 4.3 真正的商业机会在哪里

从“有需求”到“是门好生意”之间，关键不在功能多，而在于是否形成以下三个条件：

1. **高频获客入口**：免费评分、免费诊断、求职季内容流量。
2. **高意愿转化场景**：求职窗口期、投递前、面试前，用户付费意愿最强。
3. **高客单增值服务**：不仅卖模板或生成次数，而是卖“过筛概率提升”“面试准备”“岗位匹配建议”。

也就是说，这个项目如果只卖“生成一个作品集”，商业上会比较薄；但如果围绕 **评分诊断 → 重制优化 → 面试讲述 → 求职冲刺服务** 形成结果链路，就更有机会成为一个更厚的生意。

## 5. 产品形态与功能设计

### 5.1 产品形态

产品初期以 **Web 端为核心**，后续再扩展插件端。

#### A. Web 端（MVP 核心）

负责：

- 用户资料管理
- 简历导入与基础解析
- 项目导入与识别
- 项目事实补充
- AI case study 生成
- 作品集编辑
- 多格式导出
- 发布与管理

MVP 阶段优先支持以下导入方式：

- **Figma 链接导入 + OAuth 授权读取**
- 关键页面截图 / 导出图上传
- 手动补充项目结构与展示页

#### B. 设计工具插件端（后续版本）

负责：

- 在设计工具内直接读取设计稿结构
- 提取页面 / Frame / 样式 / 缩略图
- 降低导入步骤
- 提高多项目批量导入效率

### 5.2 核心输入模型

#### A. 设计师档案（长期复用，可由简历导入生成）

字段包括：

- 当前职位
- 工作年限
- 所在行业
- 擅长方向（UI / UX / 交互 / B 端 / 设计系统等）
- 当前职责范围
- 目标岗位
- 优势标签
- 表达风格偏好

#### B. 项目事实卡（每个项目一份）

字段包括：

- 项目名称
- 项目类型
- 所属行业
- 项目周期
- 个人角色
- 负责范围
- 目标用户
- 业务目标
- 关键挑战
- 关键页面
- 想突出展示的亮点
- 项目结果 / 影响

#### C. 投递目标卡（每次生成可切换）

字段包括：

- 目标岗位
- 目标公司类型
- 目标行业
- 想突出能力（视觉、系统思维、业务理解、协作推进等）

### 5.3 核心功能模块

#### 0）简历导入与信息对齐

- 支持上传 PDF / DOCX 简历，生成设计师档案初稿
- 将简历信息映射到作品集资料层，减少重复填写
- 检查简历与作品集中的职位、年限、项目名称、角色描述是否存在明显冲突
- 提供轻量美化与统一排版，输出简历增强版

#### 1）设计稿读取与项目识别

- 读取设计文件中的页面、Section、Frame、文本、色板、字体、组件信息
- 自动识别候选核心页面
- 支持用户手动勾选展示页
- 生成缩略图预览

#### 2）项目事实补充

- 引导用户补充项目背景、职责、挑战、结果
- 标注缺失信息
- 提醒用户补全影响作品集可信度的字段

#### 3）AI 结构化生成

输出内容包括：

- 项目概述
- 背景与目标
- 我的角色
- 核心页面展示
- 设计决策亮点
- 结果与价值
- 项目反思 / 经验总结

#### 4）作品集编辑器

- 可修改标题、段落、排序
- 可替换展示图
- 可隐藏不适合公开的信息
- 支持不同模板和风格方案

#### 5）多版本候选生成

默认生成 **2–3 个候选版本**，供用户选择和继续编辑。

建议首发候选方向包括：

- 国内大厂 / B 端偏专业克制版
- 通用平衡版
- C 端 / 年轻设计师偏视觉表达版

#### 6）作品集评分与诊断

- 支持用户上传已有作品集链接、PDF 或图片
- 基于质量标准输出总分与分维度评分
- 给出高层级问题提示
- 识别“视觉不过线 / 角色不清 / 结果空泛 / 投递不匹配”等核心问题
- 付费后解锁详细修改建议与平台内重制能力

#### 7）输出与发布

- 在线作品集链接
- PDF 导出
- 长图 / 封面图导出
- 后续预留站酷 / 社媒分发素材能力

#### 8）面试讲述辅助（后续阶段）

- 基于作品集内容生成项目讲述提纲
- 将 case study 转换为更适合口头表达的讲述版本
- 支持按不同面试场景生成不同讲述颗粒度：3 分钟版 / 5 分钟版 / 深问展开版

#### 9）岗位要求洞察与映射（后续阶段）

- 构建设计岗位要求知识库，覆盖初 / 中 / 高级，以及 B 端 / C 端 / 运营向 / 平面 / 品牌等方向
- 将真实招聘要求映射为作品集生成与评分约束
- 告诉用户“这个岗位更看重什么”“你当前作品集缺什么”

## 5.4 简历导入模块设计

### 5.4.1 模块目标

简历导入模块不是独立产品，而是作品集生成主链路的前置加速器，重点解决以下问题：

- 减少用户重复填写个人背景与经历信息
- 将简历作为作品集生成的事实底稿
- 让简历与作品集在职位、年限、项目名称、角色边界等信息上尽量一致
- 为后续“定向投递版本生成”提供更稳定的上下文

### 5.4.2 MVP 支持范围

MVP 阶段仅支持：

- 上传 **PDF / DOCX** 简历文件
- 解析个人基础信息、项目经历、技能标签、教育经历等结构化内容
- 生成“设计师档案初稿”
- 将简历中的项目经历映射到作品集项目库
- 对简历与作品集做基础一致性检查
- 提供轻量排版优化与导出能力

### 5.4.3 一致性检查规则

系统在生成作品集前，对简历与作品集的关键字段做一致性检查，输出“待确认项”。

#### 高优先级检查项

- 当前职位是否一致
- 工作年限是否一致
- 项目名称是否明显冲突
- 项目时间是否冲突
- 项目角色是否冲突
- 公司 / 行业背景是否冲突

### 5.4.4 与作品集生成链路的关系

- 给 AI 提供更稳定的人物背景上下文
- 减少用户在项目事实表单中的重复输入
- 让项目文案更贴近用户真实经历
- 降低简历与作品集口径不一致导致的投递风险

### 5.4.5 后续迭代方向

- 针对目标岗位的简历改写建议
- 简历与作品集的联合评分
- 针对大厂 / 创业公司 / 乙方的版本切换
- 简历 PDF 模板增强与美化
- 与项目案例页联动的“面试讲述提纲”生成

### 5.4.6 面试讲述模块设计（后续方向）

#### 模块目标

面试讲述模块的目标不是替代用户背稿，而是帮助用户围绕已生成的作品集，形成更自然、更专业的表达框架。

#### 核心输出形态

- **3 分钟项目讲解版**：适合快速自我介绍或首轮面试
- **5 分钟完整版**：适合详细 walkthrough
- **深问展开版**：围绕挑战、权衡、结果的追问准备
- **关键词卡片版**：更适合用户脱稿表达

## 5.5 MVP 用户完整流程设计

### 5.5.1 MVP 主流程目标

帮助用户在尽量短的时间内完成以下闭环：

1. 建立个人背景资料
2. 导入一个真实项目
3. 选出可展示页面
4. 补充项目事实
5. 生成作品集初稿
6. 完成基础编辑
7. 导出并用于投递

### 5.5.2 标准用户主路径

1. **注册 / 登录**
2. **创建个人档案**（上传简历或手动填写）
3. **导入项目**（Figma 链接导入 / 上传关键页面图片）
4. **确认展示页面**
5. **填写项目事实表单**
6. **选择投递目标**
7. **生成作品集初稿**
8. **编辑与修正**
9. **导出与发布**

### 5.5.3 首次用户引导策略

- 第一屏：确认用户目标（求职 / 客户展示）
- 第二屏：建立最小资料（当前职位、工作年限、目标岗位、擅长方向）
- 第三屏：先生成再补全，避免一开始表单过重

### 5.5.4 异常路径与兜底策略

- 没有简历：提供超轻量手动建档
- 没有 Figma 链接：支持图片上传或手动创建项目
- 项目保密：允许只上传部分页面，并支持打码和隐藏模块
- 觉得结果不够准：提供重生成和快速改写

## 5.6 项目事实表单字段设计

### 5.6.1 设计目标

项目事实表单是作品集生成质量的核心输入层。其目标不是让用户写一篇完整 case study，而是帮助用户以较低成本提供足够真实的项目事实，让系统基于这些信息生成可信初稿。

- 项目时间 / 周期（推荐）
- 项目阶段（推荐）
- 是否已上线（推荐）

#### 模块 2：项目背景与目标

- 项目背景（必填）
- 目标用户（必填）
- 业务目标（必填）
- 业务约束（推荐）

#### 模块 3：我的角色与职责

- 我的职位 / 角色（必填）
- 参与深度（必填）
- 负责范围（必填）
- 协作对象（推荐）
- 我的关键贡献（推荐）

#### 模块 4：关键挑战与设计判断

- 最大挑战（必填）
- 关键设计亮点（必填）

#### 模块 5：结果与价值

- 没有结果数据时的替代证据（增强）

#### 模块 6：投递导向信息

- 本次目标岗位（必填）
- 目标公司类型（推荐）

## 5.9 作品集评分系统设计

### 5.9.1 模块定位

作品集评分系统不是一个孤立的小工具，而是产品的第一层获客入口与质量控制入口。

### 5.9.2 评分系统核心目标

1. **初步可信**：评分结果不能明显违背常识
2. **足够有用**：用户能从结果中看出主要短板
3. **能够转化**：结果可以自然引导用户进入详细诊断或重制链路

### 5.9.3 评分输入方式

- 作品集链接
- PDF 作品集上传
- 图片上传

### 5.9.4 免费版输出内容

- 总分
- 等级标签（可投递 / 可优化 / 不建议直接投递）
- 8 维度分布图
- 3–5 条高层级问题摘要
- 1 个最值得优先改的问题提示

### 5.9.5 付费版输出内容

- 每个维度的分数与扣分原因
- 对应页面或模块的问题定位
- 问题严重程度分级
- 定向投递建议
- 平台内重制入口

### 5.9.6 评分结果到重制任务的映射逻辑

- 首屏专业感不足 → 切换更稳重的模板轨道
- 项目过多、深度不足 → 提示减少项目数量并重做重点 case study
- 角色不清晰 → 要求补充角色、参与深度、负责范围字段
- 缺少设计判断 → 要求补充最大挑战、为什么这样设计、权衡点
- 缺少结果证明 → 要求补充结果与价值字段
- 投递适配度不足 → 选择目标岗位并生成定向版本

## 6. MVP 范围定义

### 6.1 MVP 目标

验证以下核心假设：

1. 用户愿意从设计稿导入项目。
2. 用户愿意上传简历来自动生成基础资料，而不是重复填写个人背景。
3. 用户愿意补事实，而不是只要“全自动”。
4. AI 生成的 case study 初稿能显著降低整理时间。
5. 用户认为产出“足以拿去投递”，愿意继续编辑使用。
6. 免费评分入口能够带来稳定获客，并转化出一部分重制与订阅用户。

### 6.2 MVP 只做什么

#### 平台支持

- 首发：**Web 端 + Figma 链接导入**
- 辅助导入：关键页面截图 / 导出图上传
- 预研：即时设计、Pixso 的网页端导入路径
- 暂不首发：插件端、MasterGo 企业级集成

#### MVP 核心能力

- Web 端登录 / 项目管理
- PDF / DOCX 简历导入与基础解析
- Figma 链接导入与基础识别
- 图片上传辅助导入
- 项目事实表单
- 简历与作品集的一致性检查
- AI 生成 case study 初稿
- 模板化网页作品集生成
- PDF 导出
- 基础编辑器
- 作品集基础评分入口

#### MVP 不做什么

- 不做复杂自由排版
- 不做高度自定义建站
- 不做全自动识别所有设计亮点
- 不承诺自动生成真实数据结果
- 不做团队协作编辑
- 不做企业私有化
- 不做完整面试训练系统
- 不做大规模自动抓取招聘平台数据

## 8. 商业模式

### 8.1 当前问题判断

当前产品如果只是卖“生成作品集”或“模板 + 导出”，还不够像一门好生意，主要原因有三点：

1. **单次价值偏低**：用户可能只在求职窗口期短期使用。
2. **同类替代较多**：模板、建站工具、AI 写作工具都会稀释用户支付意愿。
3. **缺少结果导向闭环**：如果产品只负责“做出来”，但不负责“过筛、讲解、匹配岗位”，就很难抬高客单价。

因此，本项目的商业模式不应建立在“卖功能数量”上，而应建立在：

> **卖结果、卖窗口期、卖诊断与优化链路、卖对岗位的匹配度提升。**

### 8.2 更合理的收入结构

建议形成 **三层收入结构**：

#### 第一层：低门槛自助订阅收入

面向大多数用户，卖的是：

- 生成能力
- 编辑能力
- 导出能力
- 详细评分报告
- 重制与改写能力

这是基础 SaaS 收入层。

#### 第二层：高意愿求职冲刺收入

面向正在投递、面试临近、时间紧张的用户，卖的是：

- 定向诊断
- 定向重制
- 面试讲述提纲
- 岗位匹配建议
- 求职窗口期密集优化

这是更高 ARPU 的结果型收入层。

#### 第三层：服务化与 B2B 收入

面向更强需求用户或机构，卖的是：

- 人工评审 / 专家点评
- 求职训练营合作
- 求职顾问合作分发
- 企业内部设计案例沉淀 / 招聘准备工具

这是长期拉高客单和做厚利润的收入层。

### 8.3 定价方向（重构版）

#### 免费版

目标：**获客，不赚钱**。

提供：

- 1 次已有作品集基础评分
- 1 个项目导入
- 1 个作品集链接
- 基础模板与 2–3 个候选版本
- 少量基础生成次数
- 带品牌露出

免费版的核心作用不是服务完整交付，而是：

- 让用户感受到评分与诊断价值
- 让用户看到“现在这份作品集不够过线”的问题
- 引导进入付费优化链路

#### Pro 版（月付 / 年付）

目标：**服务自主型求职用户**。

提供：

- 多项目管理
- 多版本生成
- PDF 导出
- 更多模板与风格轨道
- 去品牌
- 高级改写与岗位定制
- 详细评分报告与优化建议
- 重制能力
- 自定义域名（后期）

建议定位：**自助求职工具订阅**。

#### Plus / 求职冲刺版

目标：**服务高付费意愿窗口期用户**。

提供：

- 1 个月求职冲刺套餐
- 多次改写
- AI 优化建议
- 面试展示版输出
- 作品集诊断报告
- 基于用户档案、简历与项目事实的重制服务入口
- 面试讲述提纲
- 岗位匹配建议

建议定位：**结果导向型短周期套餐**。

这类产品不一定靠长订阅挣钱，反而可以靠求职窗口期形成高客单。

#### 专家点评 / 高客单服务（后期）

目标：**提升利润厚度与品牌背书**。

形式可以包括：

- 单次专家点评
- 作品集 + 简历 + 面试讲述联合优化包
- 针对目标岗位的定向诊断
- 求职冲刺期版本管理与指导服务

这个层级的关键，不是规模最大，而是：

- 提高信任度
- 提高付费天花板
- 反哺产品训练与案例库建设

#### B2B / 企业版（后期）

目标：**补充长期稳定收入**。

提供：

- 团队作品集管理
- 企业内部案例沉淀
- 多人权限
- 私有化部署 / 专有云
- MasterGo / 国产平台深集成

### 8.4 最值得打的生意模型

如果只看“好生意”，我最看好的不是单纯 SaaS 订阅，而是：

> **免费评分获客 + Pro 自助重制订阅 + 求职冲刺高客单套餐 + 专家点评增值服务**

原因：

- 免费评分是天然流量入口。
- 重制订阅承接中等支付意愿用户。
- 求职冲刺版承接最强支付意愿窗口。
- 专家点评拉高利润和信任。

这会比“月付 39 / 59 元模板工具”更有机会成为厚生意。

### 8.5 关键经营指标建议

除了常规 SaaS 指标，更要关注这些结果型指标：

- 免费评分 → 注册转化率
- 注册 → 首次生成转化率
- 评分 → 重制链路转化率
- Pro 订阅转化率
- 求职冲刺版购买率
- 专家点评加购率
- 用户自报投递使用率
- 用户自报约面率提升感知
- 单用户求职周期内累计收入

### 8.6 商业策略建议

不把收入重心放在“平台内插件付费”，而是放在 **Web 端 SaaS + 结果导向增值服务**。

原因是：

- 各平台插件生态规则、审核和商业化机制差异较大，控制权弱。
- 作品集生成本身的支付天花板有限。
- 真正更有价值的是“诊断 + 重制 + 岗位匹配 + 面试讲述”这条更长链路。

换句话说，这个项目要从“工具收费”走向“结果收费”。

## 8.7 单位经济模型（初步假设）

### 8.7.1 说明
本节不是财务预测，而是用于判断：

> **这个项目是否有机会成为一门成立的生意，以及应该优先优化哪一个商业环节。**

以下数字均为初步经营假设，目的是帮助团队验证方向，而不是作为正式预算。

### 8.7.2 核心判断逻辑
这个项目的商业成立，不应只看订阅用户数，而应看以下三件事：

1. 免费评分是否能低成本获客
2. 评分用户是否会进入重制与订阅链路
3. 求职窗口期用户是否愿意购买高客单的冲刺服务

因此，单位经济模型建议围绕三层收入来计算：

- **免费评分获客层**
- **Pro 自助重制订阅层**
- **求职冲刺高客单层**

### 8.7.3 初步定价假设
以下为建议测试价，不是最终定价：

#### 免费版
- 价格：0 元
- 提供：1 次基础评分、1 个项目导入、基础生成能力、有限导出
- 目的：获客与激发问题意识

#### Pro 月付版
- 建议测试价：**59–99 元 / 月**
- 目标用户：自主型求职用户
- 核心卖点：详细评分、重制、改写、导出、多个项目、多版本生成

#### Pro 年付版
- 建议测试价：**299–499 元 / 年**
- 目标用户：长期经营个人作品集和职业表达的用户
- 作用：提升现金流与降低月付流失

#### 求职冲刺版
- 建议测试价：**199–399 元 / 次或 / 月**
- 目标用户：正在密集投递、面试临近、愿意为结果买单的用户
- 核心卖点：定向重制、岗位匹配建议、面试讲述提纲、快速多轮改写

#### 专家点评 / 增值服务（后期）
- 建议测试价：**399–999 元 / 次**
- 目标用户：高支付意愿、目标明确、希望获得更强信任背书的用户

### 8.7.4 用户漏斗模型（建议重点观察）
建议把经营漏斗拆为：

1. 访问 Landing Page
2. 点击“先给我的作品集打分”
3. 完成评分上传
4. 查看评分结果
5. 注册账号
6. 进入平台重制链路
7. 完成首个项目生成
8. 付费购买 Pro
9. 升级到求职冲刺版

### 8.7.5 三档转化假设
以下为用于判断商业可行性的参考假设。

#### A. 保守档
- Landing → 评分上传完成：10%
- 评分结果页 → 注册：20%
- 注册 → 首次重制：25%
- 首次重制 → Pro 付费：8%
- Pro 用户 → 冲刺版升级：8%

#### B. 中性档
- Landing → 评分上传完成：15%
- 评分结果页 → 注册：30%
- 注册 → 首次重制：35%
- 首次重制 → Pro 付费：12%
- Pro 用户 → 冲刺版升级：15%

#### C. 乐观档
- Landing → 评分上传完成：22%
- 评分结果页 → 注册：40%
- 注册 → 首次重制：45%
- 首次重制 → Pro 付费：18%
- Pro 用户 → 冲刺版升级：20%

### 8.7.6 单位收入模型（按 1000 个 Landing 访客估算）
为便于直观看生意结构，先按 1000 个访问量估算。

#### 保守档
假设：
- 1000 访客
- 100 人完成评分
- 20 人注册
- 5 人完成重制
- 0.4 人转 Pro（可按 0–1 人看）
- 0.03 人升级冲刺版

结论：
- 说明产品早期若转化不足，仅靠自然流量很难形成健康收入
- 这意味着保守档下必须继续优化评分结果页和重制链路

#### 中性档
假设：
- 1000 访客
- 150 人完成评分
- 45 人注册
- 16 人完成重制
- 约 2 人转 Pro
- 约 0.3 人升级冲刺版

若以以下价格估算：
- Pro 客单：79 元
- 冲刺版客单：299 元

则 1000 访客可带来约：
- Pro 收入：158 元
- 冲刺版收入：约 90 元
- 合计：约 248 元

这意味着：
- 如果获客成本高于这个水平，仅靠首单收入仍较吃紧
- 需要依赖复购、年付、追加项目、求职周期内多次使用来拉高 LTV

#### 乐观档
假设：
- 1000 访客
- 220 人完成评分
- 88 人注册
- 40 人完成重制
- 约 7 人转 Pro
- 约 1.4 人升级冲刺版

若按同样价格估算：
- Pro 收入：553 元
- 冲刺版收入：约 419 元
- 合计：约 972 元

这意味着：
- 一旦评分入口足够强、重制链路足够顺，产品就有机会形成更健康的获客模型
- 高客单冲刺版会显著改变整体单位经济

### 8.7.7 为什么不能只靠 Pro 订阅
从上面的假设可以看出：

- 低价月订阅虽然是基础收入层
- 但单靠月付订阅，很难支撑较高的获客成本
- 尤其用户具有明显“求职窗口期”特征，生命周期未必长

因此，本项目更合理的收入结构应是：

- **Pro 负责承接中低客单的自助型需求**
- **冲刺版负责承接高意愿、高时效的结果型需求**
- **专家点评 / 服务化收入负责拉高利润厚度**

### 8.7.8 初步 LTV 结构假设
建议把单个用户在一个求职周期内的累计价值拆成：

- 免费评分用户：0 元，但有传播和转化价值
- Pro 用户：79–199 元
- 冲刺用户：199–399 元
- 专家点评用户：399–999 元

因此可以粗略把 LTV 理解为三层：

#### 低价值用户
- 只评分、不注册或只注册不付费
- 主要贡献品牌传播与漏斗基数

#### 中价值用户
- 至少完成一次重制并购买 Pro
- 目标 LTV：**79–199 元**

#### 高价值用户
- 购买冲刺版，甚至叠加专家点评
- 目标 LTV：**300–1000 元+**

真正决定这是不是好生意的，不是平均每个人都付费，而是：

> **是否有足够比例的用户，会在求职关键窗口为结果型服务支付更高客单。**

### 8.7.9 CAC 可承受区间（初步）
可承受 CAC 不应只看 Pro 首单，而应看求职周期内 LTV。

建议按三档判断：

- 若用户主要只买 Pro，则可承受 CAC 较低
- 若用户有一定比例升级冲刺版，则 CAC 可以显著放宽
- 若后期存在专家点评等高客单服务，则可以接受更高的获客成本

初步建议：

- **纯自助 SaaS 视角**：优先把 CAC 控制在 **30–80 元** 区间内测试
- **结果型产品视角**：若冲刺版与增值服务能成立，可逐步接受 **80–200 元** 的 CAC

### 8.7.10 回本周期判断
建议重点关注两类回本模型：

#### 模型 A：自助订阅回本
- 依赖 Pro 转化和留存
- 回本更慢，但规模化更容易

#### 模型 B：求职窗口期快速回本
- 依赖评分 → 冲刺版 → 高客单服务
- 回本更快，但要求转化链路更强

对于当前项目，更值得优先验证的是：

> **评分入口是否能带出“短周期高意愿付费”模型。**

因为这比慢慢赌订阅留存更符合用户真实行为。

### 8.7.11 关键结论
基于以上假设，当前项目要成为“好生意”，至少要满足以下条件中的 2–3 个：

1. 免费评分入口能稳定带来低成本流量
2. 评分结果页能有效转化用户进入重制链路
3. Pro 转化率达到可接受水平
4. 存在一定比例用户愿意购买冲刺版
5. 后续能跑出专家点评或服务化高客单收入

如果这几项都不成立，那它更像一个“用户喜欢但不太赚钱的工具”；
如果至少有 2–3 项成立，它就有机会成长为一门结果导向型的好生意。

### 8.7.12 下一步最该验证的商业假设
建议优先验证以下 5 个假设：

1. 用户是否愿意先用“免费评分”进入产品
2. 用户是否会因为评分结果不佳而进入重制链路
3. 用户愿不愿意为“详细建议 + 重制能力”付费
4. 用户是否愿意在面试临近时购买冲刺版
5. 用户是否会把“作品集 + 讲解准备”视作一个整体付费需求

1. 用户对“免费评分 → 付费重制”的接受度有多高？

- **高客单产品必须和时效性场景绑定**，例如投递前、面试前、求职冲刺期

- 更高优先级的 AI 生成配额

适用用户：
- 正在密集投递
- 面试临近
- 对结果和效率更敏感

#### D. 专家点评 / 高客单服务（后期）
定位：高信任、高客单增值服务

建议测试价格：
- **399 元 / 599 元 / 999 元** 三档测试

建议提供：
- 专家逐项点评
- 重点项目表达建议
- 岗位匹配反馈
- 面试讲述建议
- 二次修改建议

适用用户：
- 目标非常明确
- 愿意为更高确定性买单

### 8.8.4 不同价格层卖点区隔建议
为了避免用户觉得“只是功能多一点就涨这么多钱”，每一层的卖点必须明显不同：

#### 免费版卖点
- 知道现在的问题在哪里
- 快速判断能不能投

#### Pro 版卖点
- 自己动手也能更快改好
- 有完整重制和优化能力

#### 冲刺版卖点
- 在最短时间内，把最影响投递结果的问题优先修掉
- 更适合正在投递、马上面试的人

#### 专家点评卖点
- 获得真人视角和更强信任感
- 更适合高目标岗位或高焦虑用户

### 8.8.5 优先测试的转化链路
建议先测试这三条链路，不要同时做太多复杂分支。

#### 链路 A：免费评分 → Pro 订阅
适合验证：
- 用户是否愿意为“详细建议 + 自助重制”付费

这是最基础的 SaaS 转化链路。

#### 链路 B：免费评分 → 冲刺版
适合验证：
- 用户是否会在高时效场景下跳过 Pro，直接购买结果型套餐

这条链路很重要，因为它可能直接决定你的商业天花板。

#### 链路 C：免费评分 → Pro → 冲刺版升级
适合验证：
- 用户是否愿意先低门槛进入，再在求职压力上升时升级

这条链路更接近真实用户行为。

### 8.8.6 首批实验建议
MVP 上线后，建议只跑 3 组实验，不要一口气开太多变量。

#### 实验 1：免费评分结果页 CTA 测试
测试目标：
- 看用户更容易被哪种话术推动下一步

建议对比：
- CTA A：**解锁详细建议**
- CTA B：**在平台内重制这份作品集**
- CTA C：**生成更适合大厂投递的版本**

关注指标：
- 点击率
- 注册率
- 进入重制链路比例

#### 实验 2：Pro 与冲刺版价格感知测试
测试目标：
- 看用户对不同价格梯度的接受度

建议对比：
- Pro：69 / 89 / 99 元月付
- 冲刺版：199 / 299 / 399 元

关注指标：
- 支付转化率
- 支付后首日激活率
- 用户对价格是否“值”的主观反馈

#### 实验 3：卖“工具能力”还是卖“结果”
测试目标：
- 看哪种销售叙事更容易付费

建议对比：
- 版本 A：强调“更多模板 / 多版本生成 / PDF 导出 / 去品牌”
- 版本 B：强调“更适合大厂投递 / 快速修到可投 / 面试前讲清重点”

关注指标：
- 支付转化率
- 升级率
- 用户访谈中的购买原因

### 8.8.7 支付页与方案页建议
支付页的设计不要按“功能清单”堆砌，而要按“用户此刻最关心的结果”组织。

建议结构：

#### 免费版
- 先知道问题
- 快速打分
- 看看自己作品集离可投还有多远

#### Pro 版
- 适合自己动手优化的人
- 看清问题、拿到建议、快速重制

#### 冲刺版
- 适合正在投递 / 面试临近的人
- 在最短时间内，把最影响结果的问题优先修掉

#### 专家点评
- 适合追求更高确定性的人
- 获得人工视角和更强信任背书

### 8.8.8 定价实验的成功标准
建议把实验成功定义为：

#### 免费评分入口成功
- 有足够高的上传完成率
- 用户愿意看结果页，而不是中途流失

#### Pro 成功
- 至少能形成稳定的基础付费转化
- 转化率足够支持后续内容或社区型获客

#### 冲刺版成功
- 有一部分用户明确愿意为“窗口期结果”付更高价格
- 冲刺版收入能显著改善单位经济

#### 整体成功
- 至少出现一条可持续的转化路径：
  - 评分 → Pro
  - 评分 → 冲刺
  - Pro → 冲刺

### 8.8.9 当前最推荐的商业测试顺序
建议按以下顺序推进：

1. **先验证免费评分入口是否成立**
2. **再验证评分结果页是否能推进入重制链路**
3. **再验证 Pro 是否愿意买单**
4. **随后测试冲刺版是否能承接高意愿用户**
5. **最后再考虑专家点评和更重服务化能力**

### 8.8.10 阶段性判断标准
你可以用下面这组问题，判断项目是否正在接近“好生意”：

- 免费评分是否能成为稳定获客入口？
- 用户是否真的因为评分结果产生修改动力？
- 付费用户是否把平台视为“求职结果工具”，而不是“偶尔试一下的 AI 玩具”？
- 冲刺版是否能跑出明显高于 Pro 的 ARPU？
- 是否有用户愿意进一步购买高信任服务？

如果这些问题里有 3 个以上能得到肯定答案，这个项目离“好生意”就更近了。

## 8.9 首页销售文案结构（初稿）

### 8.9.1 首页目标
首页不是展示功能大全，而是完成三个任务：

1. 让用户立刻理解：这不是普通作品集模板工具
2. 让用户立刻相信：它能帮助我更容易过筛、更容易被约面
3. 让用户立刻行动：先评分，或直接开始重制作品集

因此，首页文案必须围绕“求职结果”组织，而不是围绕“AI 能做什么”组织。

### 8.9.2 首页主叙事
建议首页主叙事为：

> **先判断你的作品集能不能投，再把它改到更能投。**

这句话能同时覆盖：
- 免费评分入口
- 作品集重制能力
- 结果导向的产品心智

### 8.9.3 首页结构建议

#### 第一屏：核心价值 + 双 CTA
目标：让用户在 5 秒内知道产品是干什么的，并给出明确入口。

建议标题：

**先给你的作品集打个分，再把它改到更能过筛。**

建议副标题：

面向国内设计师的求职表达平台。上传现有作品集或导入设计稿，快速获得评分、诊断与重制结果，帮助你更清楚地展示项目背景、角色、判断与价值。

建议 CTA：
- **先给我的作品集打分**（主 CTA）
- **直接开始生成作品集**（次 CTA）

建议辅助说明：
- 适合正在求职的 UI / UX / 产品设计师
- 支持 B 端、C 端、企业系统、运营向等项目
- 输出在线链接、PDF，并可继续编辑

#### 第二屏：痛点共鸣
目标：让用户觉得“这说的就是我”。

建议标题：

**不是没有项目，而是没有把项目讲成一份能过筛的作品集。**

建议文案点：
- 页面很多，但不知道该展示哪些
- 会做设计，但不会写 case study
- B 端 / G 端项目不够“好看”，很难包装
- 简历、作品集、面试讲述经常前后不一致
- 作品集做了很久，还是不确定到底能不能投

#### 第三屏：结果导向价值说明
目标：解释产品和普通模板工具的不同。

建议标题：

**我们不只是帮你“做一个页面”，而是帮你把求职表达做完整。**

建议用三步结构表达：

1. **先评分**
   - 看看你现有作品集距离“可投递”还有多远

2. **再重制**
   - 基于简历、设计稿和项目事实，生成更适合投递的作品集版本

3. **再优化讲述**
   - 后续可继续生成项目讲述提纲，帮助你在面试里把作品集讲清楚

#### 第四屏：评分能力展示
目标：突出免费入口的吸引力。

建议标题：

**先上传你现在的作品集，我们会告诉你它卡在哪。**

建议展示内容：
- 总分示意
- 8 个维度评分示意
- 问题摘要示意，例如：
  - 首屏专业感不足
  - 看不出你的具体贡献
  - 缺少结果与价值证明
  - 更像项目展示，不像求职 case study

建议 CTA：
- **免费获取基础评分**

#### 第五屏：重制能力展示
目标：让用户理解付费价值。

建议标题：

**不是只告诉你哪里不行，而是直接帮你改到更能投。**

建议展示内容：
- 导入简历
- 导入 Figma / 页面图片
- 补充项目事实
- 一次生成 2–3 个候选版本
- 按目标岗位重制内容表达

建议强调：
- 不只是换模板
- 而是把“项目背景、角色、判断、结果”重新讲清楚

#### 第六屏：适用人群与场景
目标：提高代入感和转化率。

建议标题：

**尤其适合这些设计师。**

建议分组展示：
- **正在找工作的 UI / UX / 产品设计师**
- **做 B 端 / G 端 / 企业系统的设计师**
- **项目很多，但一直没整理作品集的人**
- **作品集能看，但总觉得还不够过筛的人**
- **想把作品集和简历、面试表达统一起来的人**

#### 第七屏：差异化说明
目标：回答“为什么不用模板站 / 建站工具”。

建议标题：

**你不是缺一个建站工具，你是缺一个更懂求职筛选逻辑的表达工具。**

建议对比逻辑：
- 普通模板工具：更偏页面搭建
- 本产品：更偏求职表达、评分诊断、重制优化、岗位匹配

可以用 3 列对比：
- 模板工具
- 通用 AI 写作工具
- 作品集工作台

#### 第八屏：付费方案概览
目标：为后续商业化做铺垫。

建议标题：

**先免费评分，再选择适合你的优化方式。**

建议展示：
- **免费版**：先知道问题在哪里
- **Pro**：适合想自己快速改好的人
- **求职冲刺版**：适合正在投递、想在最短时间内修到更能投的人

注意：
- 首页不需要把所有价格细节一次讲完
- 更重要的是讲清楚每一层“解决什么问题”

#### 第九屏：信任建立
目标：减少“这是 AI 玩具”的疑虑。

建议展示内容：
- 示例作品集前后对比
- 同一项目不同版本示例
- B 端项目如何被讲得更清楚的示例
- 用户反馈（后续补）
- 评分维度来源与方法说明（后续补）

#### 第十屏：底部强 CTA
目标：促成最终行动。

建议标题：

**先看看你现在的作品集，离“可投递”还有多远。**

建议 CTA：
- **免费给我的作品集打分**
- **直接开始重制作品集**

### 8.9.4 首页文案语气建议
首页文案整体不建议写得太像“AI 神器”，而应更像：

- 专业
- 稳定
- 有招聘语境
- 不夸张承诺

建议避免的表达：
- 一键拿面试
- 自动生成高薪作品集
- 秒速搞定大厂 offer

建议坚持的表达：
- 更能投
- 更接近初筛要求
- 更容易讲清楚
- 更贴近岗位期待

### 8.9.5 首页首屏文案备选

#### 版本 A：偏结果导向
**先给你的作品集打个分，再把它改到更能过筛。**

上传现有作品集，快速获得评分、诊断与重制建议。也可以直接导入设计稿，生成更适合国内设计岗位投递的作品集版本。

#### 版本 B：偏用户痛点
**不是没有项目，而是没有一份真正能过筛的作品集。**

我们帮你把设计稿、简历和项目事实整理成更适合投递的 case study，并告诉你当前作品集还差什么。

#### 版本 C：偏求职语境
**给设计师的求职表达工作台。**

先判断作品集能不能投，再把项目背景、个人角色、设计判断和结果价值讲清楚。

### 8.9.6 首页最值得优先测试的 3 组文案
建议上线初期只测试这三组，不要同时改太多变量。

#### 测试 1：主标题方向
- A：先给你的作品集打个分，再把它改到更能过筛
- B：不是没有项目，而是没有一份真正能过筛的作品集
- C：给设计师的求职表达工作台

#### 测试 2：主 CTA 文案
- A：先给我的作品集打分
- B：免费查看作品集得分
- C：看看我的作品集还差什么

#### 测试 3：次 CTA 文案
- A：直接开始生成作品集
- B：导入项目并重制作品集
- C：生成更适合投递的版本

### 8.9.7 当前最推荐的首页版本
如果现在就要先做一版首页，我最推荐：

#### 标题
**先给你的作品集打个分，再把它改到更能过筛。**

#### 副标题
面向国内设计师的求职表达平台。上传现有作品集或导入设计稿，快速获得评分、诊断与重制结果，帮助你更清楚地展示项目背景、角色、判断与价值。

#### 主 CTA
**先给我的作品集打分**

#### 次 CTA
**直接开始生成作品集**

这个版本最适合当前阶段，因为它同时承接了：
- 免费评分入口

- 结果导向的产品定位
- 后续商业转化空间

## 10. 获客策略

### 10.1 冷启动获客

- 免费版生成后带品牌水印

- 免费评分后提示“解锁详细建议”

- 提供“再生成一个岗位版本”的自然升级点

- 设计社区合作

- 设计工具插件市场上架

- 校招求职群体合作投放

## 11. 风险与应对

### 11.1 风险：AI 生成内容与真实项目有偏差

**应对：**

- 用“事实输入层”约束生成
- 对不确定内容不写死为事实
- 所有关键段落可编辑
- 显示“缺失信息提醒”

### 11.2 风险：网页端导入后识别不准

**应对：**

- 优先采用结构化导入（Figma 链接授权）而不是只依赖静态图片
- 自动识别 + 用户手动确认双机制
- 支持拖拽重排与筛选
- 对图片上传场景降低自动化预期，转为“辅助生成”模式

### 11.3 风险：用户觉得还是要改很多

**应对：**

- 先追求 80 分初稿，而不是一步到位
- 优化“改写”和“补充信息”流程
- 聚焦可用性，不追求复杂建站
- 先把简历导入做成“自动填表 + 一致性检查”，而不是一开始就做成复杂简历编辑器

### 11.4 风险：评分看起来有价值，但不形成付费

**应对：**

- 评分结果页必须直接对应到可执行动作
- 不能只停留在“你的作品集 67 分”这类浅层反馈
- 让用户清楚知道进入平台后能改掉哪些问题
- 明确免费与付费的边界：免费给判断，付费给方案和重制

### 11.5 风险：商业模式过于依赖低价订阅

**应对：**

- 不把主要收入押在低价月订阅上
- 尽早验证求职冲刺版和高客单增值服务
- 把高支付意愿窗口期作为核心经营场景

### 11.6 风险：平台策略变化

**应对：**

- 核心价值放在 Web 端，不绑死某一平台
- 优先做插件数据导入，不依赖深度私有 API
- 多平台适配策略前置设计

### 11.7 风险：用户担心隐私和保密

**应对：**

- 支持页面选择性同步
- 支持打码 / 隐藏敏感模块
- 默认不公开未发布项目
- 企业版预留私有化方案

### 11.8 风险：被平台型产品挤压

**应对：**

- 不正面竞争“建站”
- 聚焦“国内求职语境 + 复杂项目包装 + 真实事实驱动 + 求职结果链路”
- 强化评分、重制、岗位匹配、面试讲述这些更结果导向的能力

## 12. 成功指标

### 12.1 北极星指标

**7 天内生成并保留的“可投递作品集”数量**

定义：用户完成导入、补事实、生成、编辑并保留链接 / 导出 PDF 的项目数量。

### 12.2 核心产品指标

- 项目导入完成率
- 项目事实填写完成率
- 首次生成完成率
- 首次生成到首次编辑率
- 作品集评分入口使用率
- 评分后进入重制链路的转化率
- 导出率（PDF / 链接）
- 7 日留存
- 多项目生成率
- 付费转化率

### 12.3 商业指标

- 免费评分 → 注册转化率
- 注册 → 首次生成转化率
- Pro 订阅转化率
- 求职冲刺版购买率
- 专家点评加购率（后期）
- 单用户求职周期内累计收入
- LTV / CAC

### 12.4 质量指标

- 用户主观评分：这份作品集是否“敢拿去投递”
- 用户主观评分：是否比手工从零开始省时
- 平均生成后编辑次数
- 用户愿意分享 / 推荐比例

### 12.5 结果指标（后期）

- 用户反馈的投递使用率
- 获得约面 / 初筛通过的自报比例
- 求职周期内复购 / 多版本生成次数

## 13. 当前建议的产品边界

### 该做

- 设计稿读取
- 事实补充
- case study 初稿生成
- 可投递模板
- 链接 + PDF 输出
- 岗位定向改写
- 作品集评分与基础诊断
- 结果页到重制链路设计

### 暂不做（MVP 阶段）

- 自由建站
- 高级交互动效编辑
- 复杂团队协同
- 全自动写出所有业务指标
- 代替用户完成完整面试训练
- 自动大规模抓取招聘平台数据

### 中长期扩展方向

- 面试讲述提纲与追问准备
- 岗位要求知识库
- 简历 + 作品集 + 面试讲述联动优化
- 求职冲刺服务
- 专家点评服务
- 设计师就业指导服务

## 14. 结论

这是一个 **值得做，但必须从“工具思维”走向“结果思维”** 的项目。

它最有机会赢的，不是“自动化程度”本身，而是：

- 是否真的理解国内设计师的投递场景
- 是否能把复杂项目包装成更容易过筛的 case study
- 是否能在 10–20 分钟内产出一份用户愿意继续修改并投出去的初稿
- 是否能通过评分、重制、岗位匹配、面试讲述，形成更厚的商业链路

现阶段最合理的路线是：

1. **以求职设计师为核心用户**
2. **以免费评分作为获客入口**
3. **以作品集重制作为核心转化能力**
4. **以求职冲刺版和增值服务提升客单价**
5. **后续向岗位知识库与面试讲述模块延伸**

这份产品如果只卖“生成一个作品集”，很难成为好生意；但如果它能持续帮助设计师 **更容易过筛、更容易讲清、更贴近岗位要求**，它就有机会从一个工具成长为一门更厚的生意。
